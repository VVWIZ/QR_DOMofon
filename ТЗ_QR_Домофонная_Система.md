## 1. Общая архитектура системы

### 1.1 Назначение системы

Система предназначена для организации доступа в жилые комплексы, дома, подъезды, калитки и на территорию (шлагбаумы) с использованием QR-кодов и мобильных устройств, с поддержкой видеозвонков, удалённого открытия дверей и детального аудита событий.

Система проектируется как коммерческий, массовый продукт с целевой нагрузкой до **20 млн пользователей** и **~90 000 домов**.

### 1.2 Архитектурный подход

Архитектура — распределённая, отказоустойчивая, ориентированная на горизонтальное масштабирование.

**Основные принципы:**
- Stateless backend-сервисы
- Явное разделение control-plane и media-plane
- Fail-open логика для физических устройств доступа
- Централизованный аудит и логирование
- Multi-tenant изоляция на уровне данных и API

### 1.3 Основные компоненты системы

#### 1.3.1 Клиентские компоненты

| Компонент | Платформа | Пользователь |
|-----------|-----------|--------------|
| Мобильное приложение жильца | iOS / Android | Жилец, владелец квартиры |
| Веб-клиент посетителя | Browser | Посетитель (QR-страница) |
| Веб-клиент гостя | Browser | Гость (ссылка без приложения) |
| Веб-админка УК | Browser | Управляющая компания |
| Веб-админка сисадмина | Browser | Системный администратор |

#### 1.3.2 Серверные компоненты

| Сервис | Назначение |
|--------|-----------|
| API Gateway | Маршрутизация, rate limiting, JWT-валидация |
| Auth Service | Аутентификация, управление токенами, инвайт-ссылки |
| User & RBAC Service | Профили пользователей, роли, права |
| Property Service | ЖК, дома, подъезды, квартиры, AccessPoint |
| Access Control Service | Проверка прав, выдача команд на устройства |
| QR Service | Генерация, валидация, ротация QR-кодов |
| Call Signaling Service | WebRTC signaling (WebSocket) |
| Media SFU | WebRTC media forwarding |
| Snapshot Service | Фотофиксация событий |
| Notification Service | Push (FCM/APNS), WhatsApp/Telegram уведомления |
| Audit & Logging Service | Immutable audit trail |
| Device Registry | MQTT-брокер, реестр устройств, OTA |

#### 1.3.3 Интеграционные компоненты

- GSM/GPRS устройства (ESP32 / Arduino + реле) — подключение через MQTT
- Push-провайдеры: FCM (Android), APNS (iOS)
- Мессенджеры: WhatsApp Business API, Telegram Bot API
- Облачное хранилище: S3-compatible (фото, видеообращения)
- MQTT-брокер (EMQ X / Mosquitto clustered)

### 1.4 Потоки данных (high-level)

```
Посетитель сканирует QR-код
  → Открывается веб-страница (привязана к AccessPoint)
  → GPS-проверка (≤50 м)
  → Посетитель выбирает квартиру
  → Инициируется WebRTC-звонок (browser → SFU → mobile)
  → Жилец принимает звонок
  → Жилец разрешает доступ → команда в Access Control Service
  → Access Control Service публикует команду в MQTT
  → ESP32/Arduino подписан на топик → подаёт сигнал 5 В на катушку реле → реле размыкает силовую цепь замка (12–15 В) → замок обесточен → дверь открыта
  → Snapshot Service фиксирует фото
  → Audit Service логирует событие
```

### 1.5 Trust Boundaries

| Компонент | Уровень доверия | Примечание |
|-----------|----------------|------------|
| Веб-клиент посетителя | Недоверенный | Все решения на сервере |
| Веб-клиент гостя | Недоверенный | Проверка времени доступа на сервере |
| Мобильное приложение | Условно доверенное | JWT + refresh token |
| Веб-админка УК | Условно доверенное | JWT + RBAC на сервере |
| ESP32/Arduino устройства | Минимально доверенные | MQTT + TLS + device cert |
| Внутренние сервисы | Доверенные | mTLS между сервисами |

Все критические решения (открытие двери, проверка прав) принимаются **только сервером**.

### 1.6 Latency требования

| Операция | Требование |
|----------|-----------|
| Инициация звонка | ≤ 2 секунды |
| Открытие двери после подтверждения | ≤ 2 секунды |
| Задержка WebRTC-медиа | ≤ 500 мс |
| Доставка MQTT-команды на устройство | ≤ 1 секунда |
| Ответ REST API (p95) | ≤ 300 мс |

### 2.1 Иерархия недвижимости

```
Жилой комплекс (ResidentialComplex)
└── Дом (Building)
```

## 3. Роли и модель доступа (RBAC)

### 3.1 Роли системы

#### 3.1.1 SystemAdmin (Системный администратор)
- Полный доступ ко всей системе
- Управление УК (создание, отключение)
- Доступ ко всем логам и метрикам
- Управление устройствами и прошивками

#### 3.1.2 ManagementCompanyAdmin (Администратор УК)
- Доступ **только** к своим ЖК и домам (tenant isolation)
- Управление домами, подъездами, калитками, шлагбаумами
- Назначение владельцев квартир
- Принудительный отзыв любых доступов (с логированием)
- Просмотр логов калиток и шлагбаумов своих домов
- **Нет доступа** к логам звонков в квартиры

#### 3.1.3 ApartmentOwner (Владелец квартиры)
- Управление всеми доступами своей квартиры
- Добавление жильцов (с выбором уровня прав)
- Удаление жильцов
- Просмотр всей истории событий квартиры

#### 3.1.4 Resident (Жилец)
- Приём входящих звонков
- Открытие дверей (входных групп, к которым имеет доступ)
- Создание гостей (в пределах лимитов квартиры)
- Создание курьерских доступов
- Просмотр истории звонков своей квартиры

#### 3.1.5 Guest (Гость)
- Временный доступ по ссылке или QR
- Доступ ограничен по времени (максимум 2 дня)
- Доступ ограничен конкретными AccessPoint
- Нет аккаунта в системе — аутентификация через временный токен ссылки
- Срок > 2 дней → следует добавить как Resident

#### 3.1.6 Courier (Курьер)
- Одноразовый или временный доступ (максимум 24 часа)
- Ограничен цепочкой AccessPoint (напр.: шлагбаум → калитка → подъезд)
- Аутентификация через временный токен ссылки

### 3.2 Матрица доступа (ключевые операции)

| Операция | SysAdmin | УК | Owner | Resident | Guest | Courier |
|----------|----------|----|-------|----------|-------|---------|
| Открыть дверь подъезда | ✓ | — | ✓ | ✓ | ✓* | ✓* |
| Открыть калитку | ✓ | ✓ | ✓ | ✓ | ✓* | ✓* |
| Открыть шлагбаум | ✓ | ✓ | ✓ | ✓ | — | ✓* |
| Принять звонок | ✓ | — | ✓ | ✓ | — | — |
| Создать гостя | ✓ | — | ✓ | ✓‡ | — | — |
| Отозвать доступ | ✓ | ✓† | ✓ | ✓§ | — | — |
| Просмотр логов квартиры | ✓ | — | ✓ | ✓ | — | — |
| Просмотр логов калитки/шлагбаума | ✓ | ✓† | — | — | — | — |
| Управление домами | ✓ | ✓† | — | — | — | — |
| Управление УК | ✓ | — | — | — | — | — |

*только в пределах выданных прав и временного окна  
†только для своих объектов  
‡если Owner делегировал право (`can_create_guests`)  
§только доступы (гости/курьеры), созданные самим жильцом

### 3.3 Особенности RBAC

- Один пользователь может иметь несколько ролей в разных квартирах (напр.: Resident в кв.12 и Owner в кв.45)
- Роли изолированы контекстом квартиры / ЖК
- Все проверки выполняются **сервером** при каждом запросе
- Делегирование прав: Owner может наделить Resident правом создавать гостей

## 4. Пользовательские сценарии

Все сценарии считаются **критичными** и обязательными для реализации в MVP.

### 4.1 Посетитель сканирует QR и звонит в квартиру

**Участники:** Посетитель (браузер), QR Service, Property Service, Call Signaling Service, Mobile App (жильцы)

```
1.  Посетитель сканирует QR-код на двери подъезда или калитке
2.  Браузер открывает HTTPS-страницу: https://app.example.com/v?aid={public_id}&v=1&sig={signature}
3.  QR Service валидирует public_id и HMAC-подпись
4.  Браузер запрашивает разрешение GPS
5.  Веб-клиент отправляет координаты на сервер
6.  Сервер проверяет геопозицию (радиус ≤ 50 м от AccessPoint.gps_lat/lon)
7.  При неуспешной проверке — страница показывает ошибку, дальнейшие действия заблокированы
8.  При успехе — посетитель **вводит номер квартиры вручную** (основной режим — не раскрывает заселённость). Список квартир показывается только если у дома включён публичный справочник и лишь для квартир с `is_listed = true` (opt-out жильца, §2.2.4)
9.  Посетитель выбирает/подтверждает квартиру
10. Call Signaling Service создаёт WebRTC-сессию
11. Snapshot Service начинает фотофиксацию
12. Push-уведомления отправляются всем жильцам квартиры одновременно
13. Любой жилец может принять или отклонить звонок
```

**Таймаут ожидания ответа:** 30 секунд (настраиваемое)

### 4.2 Приём видеозвонка жильцом

```
1.  Жильцу приходит push-уведомление (звук + вибрация)
2.  В уведомлении: название ЖК, номер подъезда/калитки, превью фото посетителя
3.  Жилец открывает экран звонка в приложении
4.  Snapshot Service продолжает фотофиксацию
5.  По умолчанию: видео от посетителя активно, аудио двустороннее
6.  Жилец может отключить отображение видео локально
7.  Жилец может:
    - Нажать «Открыть» — открыть дверь без завершения звонка
    - Нажать «Отклонить» — завершить сессию без открытия
    - Завершить разговор вручную
```

### 4.3 Открытие двери по результату звонка

```
1.  Жилец нажимает кнопку «Открыть» в приложении
2.  Mobile App отправляет POST /access/open с JWT
3.  Access Control Service проверяет:
    - валидность JWT
    - наличие у пользователя права открывать данный AccessPoint
    - активность сессии звонка
4.  Access Control Service публикует MQTT-команду: devices/{device_id}/commands
5.  Device подписан на топик → получает команду → подаёт сигнал 5 В на катушку реле → реле размыкает силовую цепь замка (12–15 В) → замок обесточен → дверь открыта
6.  Сигнал на реле удерживается 5 секунд (настраиваемое), затем снимается → замок снова под питанием → дверь заперта
7.  Device публикует подтверждение: devices/{device_id}/status
8.  Audit Service логирует событие с метаданными
```

### 4.4 Никто не ответил на звонок

```
1.  По истечении таймаута (30 сек) — звонок завершается автоматически
2.  Посетителю предлагается оставить видеообращение
3.  Посетитель записывает видео (до 60 секунд)
4.  Видеообращение загружается в S3-compatible хранилище
5.  Жильцам квартиры отправляется push-уведомление со ссылкой на видео
6.  Срок хранения видеообращения: 7 дней, затем автоудаление по TTL
7.  Доступ к видеообращению: только жильцы квартиры
```

> **Примечание:** Видеообращения хранятся 7 дней (не «без хранения»). Хранение необходимо для доставки жильцам.

### 4.5 Удалённое открытие двери без звонка

```
1.  Жилец открывает мобильное приложение
2.  На главном экране — список доступных адресов и AccessPoint
3.  Жилец выбирает нужную входную группу (подъезд / калитка / шлагбаум)
4.  Нажимает кнопку открытия
5.  Запрос проходит валидацию в Access Control Service
6.  MQTT-команда отправляется на устройство
7.  Событие логируется
```

### 4.6 Гость получает доступ по ссылке

```
1.  Жилец создаёт гостя: указывает имя, период доступа (до 2 дней), доступные AccessPoint
2.  Система генерирует временную ссылку: https://app.example.com/g/{token}
3.  Жилец пересылает ссылку гостю через мессенджер
4.  Гость переходит по ссылке — открывается веб-страница без регистрации
5.  На странице — кнопки открытия доступных входных групп
6.  При нажатии:
    - Сервер проверяет token (срок, количество использований, список AccessPoint, не отозван)
    - Сервер отправляет MQTT-команду на устройство
    - Событие логируется с привязкой к гостевому токену
7.  По истечении срока — токен инвалидируется, страница показывает ошибку
```

> **Модель угроз и митигации (гостевые/курьерские ссылки).** Ссылка — bearer-токен: любой, у кого она есть, может открыть дверь удалённо (пересылается через мессенджер, может утечь). Для входа в подъезд это согласуется с принятой политикой fail-open (§10.1) — сам вход не является высокобарьерным. Тем не менее заложены митигации:
> - короткий TTL (гость ≤ 2 дня, курьер ≤ 24 ч), одноразовый режим для курьеров (`is_one_time`);
> - лимиты активных доступов (§8.2, §8.3), досрочный отзыв владельцем/жильцом и принудительный — УК (§8.5);
> - каждое открытие по токену логируется в аудит (§16);
> - **опционально** (флаг на AccessPoint): требовать гео-проверку (≤ радиус точки) и/или одноразовый код у двери для открытия по гостевой ссылке — включается для точек с повышенными требованиями (напр. шлагбаум паркинга).
>
> ✅ **Решение (2026-07-06):** риск bearer-ссылки без обязательной проверки присутствия для подъездов **принят** (симметрично политике fail-open, §10.1). Обязательная гео/код-проверка остаётся опциональным флагом на AccessPoint для точек с повышенными требованиями.

### 4.7 Открытие шлагбаума из приложения

```
1.  УК назначает пользователю доступ к шлагбауму (тип AccessPoint = barrier)
2.  В приложении пользователя появляется кнопка соответствующего шлагбаума
3.  Пользователь нажимает кнопку
4.  Access Control Service проверяет права
5.  MQTT-команда отправляется на устройство
6.  Событие логируется
```

## 5. QR-домофон

### 5.1 Типы QR-кодов

| Тип | Применение | MVP |
|-----|-----------|-----|
| Статический | Основной тип. Размещается на двери подъезда или калитке физически | ✓ |
| Динамический | Генерируется сервером с TTL, обновляется по расписанию | Архитектурно заложить |

**Для шлагбаумов QR-коды не используются.** Управление только через приложение.

### 5.2 Привязка QR-кода

Связь строго **1:1**: `QRCode → AccessPoint → Device`

- Один QR-код — один объект доступа
- QR-код не содержит чувствительных данных
- Без серверной валидации QR-код не даёт никакого доступа

### 5.3 Формат и подпись QR-кода

**URL в QR-коде:**
```
https://app.example.com/v?aid={public_id}&v=1&kid={key_id}&sig={signature}
```

| Поле | Описание |
|------|---------|
| `aid` | `AccessPoint.public_id` (UUID, не внутренний id) |
| `v` | Версия схемы (текущая: 1) |
| `kid` | ID ключа, которым подписан QR (для ротации без перепечатки) |
| `sig` | HMAC-SHA256 подпись от `aid + v + kid`, ключ — секрет сервиса QR Service |

**Алгоритм подписи:**
```
message = aid + ":" + v + ":" + kid
signature = HMAC-SHA256(message, QR_SECRET_KEY[kid])
sig = base64url(signature)[0:32]  // первые 32 символа base64url
```

**Ротация ключа подписи (без перепечатки статических QR):**
- Сервис хранит **набор активных ключей**, каждый со своим `kid`. Валидация выбирает ключ по `kid` из URL.
- Ротация вводит **новый** `kid` для вновь генерируемых QR; **старые `kid` остаются валидными** до истечения их срока → ранее напечатанные статические QR продолжают работать.
- Перепечатка статических QR требуется только при **компрометации** конкретного ключа (тогда его `kid` отзывается) — не по расписанию.
- Динамические QR регенерируются автоматически на актуальном `kid`.

> **Примечание по модели угроз:** `aid` и `sig` напечатаны на физическом QR и публичны. Подпись защищает от подделки URL для произвольного `public_id` без знания секрета, но не заменяет серверную валидацию, гео-проверку и одноразовый session-token (§5.6). Массовая плановая ротация ключа как мера безопасности для статических QR избыточна и заменена отзывом по компрометации.

### 5.4 Поведение при сканировании

```
1.  Пользователь сканирует QR → браузер открывает URL
2.  QR Service валидирует:
    a. Формат URL (наличие aid, v, sig)
    b. HMAC-подпись
    c. AccessPoint активен (is_active = true)
    d. Устройство подключено (Device.status = online)
       — Если устройство offline: показывается предупреждение «Устройство временно недоступно, но дверь может быть открыта. Вы можете позвонить жильцу.» Звонок не блокируется — жилец может открыть дверь вручную.
3.  Браузер запрашивает разрешение на GPS
4.  Клиент отправляет координаты
5.  Сервер выполняет геопроверку
6.  При любой ошибке — показывается конкретное сообщение об ошибке
7.  При успехе — список квартир для звонка
```

### 5.5 Геопозиционная проверка

- Источник: GPS браузера (Geolocation API)
- Радиус допуска: **50 метров** (настраиваемое на уровне AccessPoint)
- При отказе пользователя от GPS — звонок заблокирован
- Геопроверка может быть отключена администратором для конкретного AccessPoint (флаг `gps_check_enabled`)

> **Статус контроля:** геопроверка — **defense-in-depth, а не гарантия присутствия**. Браузерный GPS (а) спуфится (DevTools/расширения подменяют координаты) и (б) неточен в городе/у двери (погрешность может превышать радиус → ложные отказы легитимным визитёрам). Поэтому:
> - геопроверка не является единственным барьером — критичен серверный контроль прав, одноразовый session-token и то, что решение об открытии принимает **жилец** во время звонка;
> - при большой погрешности GPS (accuracy хуже радиуса) показывать понятную ошибку и путь «повторить/ввести номер квартиры вручную», а не молча блокировать;
> - на точках с высокими требованиями геопроверку можно ужесточать, на прочих — ослаблять/выключать (`gps_check_enabled`).

### 5.6 Защита от злоупотреблений

| Мера | Параметр |
|------|---------|
| Rate limiting по IP | 10 сканирований / минута |
| Rate limiting по public_id | 30 сканирований / минута |
| Rate limiting звонков | 5 звонков / 10 минут — по паре (IP + session_token), не только по IP |
| Защита от replay атак | Session token одноразовый, TTL 5 минут |
| Валидация GPS | Координаты проверяются на сервере |

> **Примечание по IP-лимитам:** ограничение только по IP даёт ложные срабатывания за CGNAT/корпоративным NAT (много абонентов = один IP) и обходится ротацией IP. Поэтому лимиты комбинируются с ключами `public_id` и одноразового session-token; при срабатывании — CAPTCHA/задержка, а не жёсткая блокировка легитимных пользователей.

### 5.7 Dynamic QR (архитектурная закладка)

Для будущей поддержки динамических QR предусмотреть:
- Поле `expires_at` в таблице `QRCode`
- Job-планировщик для ротации динамических QR по расписанию
- API-эндпоинт генерации нового QR для AccessPoint
- Обратная совместимость: статические QR проверяются без поля TTL

## 6. WebRTC и видеочасть

### 6.1 Общий подход

Видеосвязь реализуется на базе WebRTC по модели:

```
Browser (Visitor) → SFU → Mobile App (Residents)
```

Видео передаётся **только от посетителя к жильцам**. Аудио — двустороннее.

### 6.2 Signaling

- Реализуется через **Call Signaling Service**
- Протокол: **WebSocket** (постоянное соединение)
- Сервер участвует только в signaling, не в медиапотоке
- SDP offer/answer обмен через signaling сервер
- ICE candidates: Trickle ICE

**Жизненный цикл сессии:**
```
Visitor  →  [ws connect]  →  Signaling Service
Signaling Service  →  [push notify]  →  Resident Mobile App
Mobile App  →  [ws connect]  →  Signaling Service
Visitor + Resident  →  [SDP exchange via Signaling]
Visitor + Resident  →  [ICE via STUN/TURN]  →  [always via SFU]
```

### 6.3 Media Plane

- Используется **SFU** (Selective Forwarding Unit)
- Рекомендуемые решения: **LiveKit** или **Mediasoup** (выбор команды)
- Поддержка горизонтального масштабирования (cluster mode)
- Ограничение входящего битрейта с браузера: 500 Kbps
- Разрешение: до 720p

### 6.4 Управление видео

- Видео от посетителя включено всегда, посетитель **не может отключить**
- Жилец может отключить отображение видео локально (не влияет на стрим)
- Аудио от жильца и посетителя — двустороннее

### 6.5 Видеообращения

- Запись выполняется на стороне браузера (MediaRecorder API)
- Формат: WebM (VP8/Vorbis) или MP4 (H.264/AAC) в зависимости от браузера
- Максимальная длительность: **60 секунд**
- После записи — загрузка в S3-compatible хранилище
- **Срок хранения: 7 дней**, затем автоудаление по TTL
- Жильцам отправляется push-уведомление со ссылкой на просмотр
- Доступ: только жильцы квартиры, по временной pre-signed URL (TTL 24 часа)

### 6.6 TURN / STUN

- STUN — обязателен (Google STUN или self-hosted)
- TURN — fallback при невозможности P2P (корпоративные сети, симметричный NAT)
- TURN-серверы масштабируются горизонтально за load balancer
- Протокол TURN: UDP/TCP, порт 3478 / TLS 5349
- **Аутентификация TURN — временные креды** (coturn REST API / `use-auth-secret`): username с TTL + HMAC-пароль, выдаются сервером на сессию. Постоянные статические логины не используются (защита от абуза релея посторонним трафиком)

### 6.7 Ограничения

| Параметр | Значение |
|---------|---------|
| Максимальная длительность звонка | 60 секунд (настраиваемое; проверить как значение по умолчанию — для реального разговора у двери может быть мало, рассмотреть 120 сек) |
| Таймаут ожидания ответа | 30 секунд (настраиваемое) |
| Макс. одновременных соединений к одной квартире | Не ограничено логически |
| Битрейт входящего видео | ≤ 500 Kbps |
| Разрешение | До 720p |

## 7. Фотофиксация и хранение данных

### 7.1 Назначение

Фотофиксация используется для аудита событий, повышения безопасности и последующего анализа инцидентов.

### 7.2 Алгоритм фотофиксации

| Момент | Действие |
|--------|---------|
| Начало звонка (первые 5 сек) | 3 фотографии |
| Во время звонка | 1 фото каждые 5 секунд |
| При завершении звонка | 1 итоговое фото |

Фотофиксация выполняется **независимо** от того, был ли звонок принят.  
Источник фото: видеопоток браузера посетителя (getUserMedia).

> **Отказ в доступе к камере:** без разрешения на камеру видеозвонок и фотофиксация невозможны технически. Поведение: до звонка запрашивается доступ к камере; при отказе — показать сообщение, что видеозвонок недоступен, и предложить аудио-режим (если допускается политикой точки) либо оставить видеообращение. Событие `snapshot_error`/`camera_denied` логируется. Согласие на обработку (§21.2) запрашивается до запроса камеры.

### 7.3 Метаданные фото

```json
{
  "timestamp": "2026-05-26T10:23:45Z",
  "access_point_id": "uuid",
  "apartment_id": "uuid",
  "call_session_id": "uuid",
  "event_type": "call_start | call_frame | call_end",
  "user_id": "uuid (если применимо)"
}
```

### 7.4 Хранение

- Хранилище: S3-compatible (MinIO, AWS S3, или аналог)
- Бакет: приватный, без публичного доступа
- Срок хранения: **30 дней**, автоудаление по TTL (S3 Lifecycle Policy)
- Доступ через pre-signed URL (TTL 1 час)

### 7.5 Доступ к данным

| Событие | Кто видит фото |
|---------|---------------|
| Звонок в квартиру | Только жильцы квартиры |
| Звонок на калитку | Жильцы + администратор УК |
| Событие на шлагбауме | Жильцы + администратор УК |

## 8. Управление доступами

### 8.1 Типы доступов

| Тип | Описание |
|-----|---------|
| Постоянный | Для жильцов, без срока истечения |
| Временный | Для гостей и курьеров, с датой начала и конца |
| Одноразовый | Только для курьеров, аннулируется после использования |

### 8.2 Гости

- Максимум **15 активных гостей** на квартиру одновременно
- Максимальный срок доступа: **2 дня**
- Если нужен доступ дольше 2 дней → добавить человека как Resident (жилец)
- Доступ отзывается в любой момент: владельцем (любой доступ квартиры) или жильцом (только созданные им доступы). УК — принудительно (§8.5)
- Аутентификация гостя — через временную ссылку, аккаунт не требуется

### 8.3 Курьеры

- Доступ: одноразовый или временный (максимум 24 часа)
- Ограничен цепочкой AccessPoint (напр.: шлагбаум → калитка → подъезд)
- Аутентификация — через временную ссылку, аккаунт не требуется
- Лимит активных курьерских доступов: **5 на квартиру**

### 8.4 Цепочки доступа

Курьерам и гостям может быть выдан доступ к нескольким AccessPoint последовательно:

```
Пример цепочки: [шлагбаум_A] → [калитка_B] → [подъезд_C]
```

- Доступ ко всем AccessPoint в цепочке открывается одним токеном
- Порядок использования не контролируется системой (только права) — курьер может открыть подъезд, не пройдя шлагбаум. ✅ **Решение (2026-07-06):** порядок-контроль **не вводится** (упрощение); при будущей необходимости — отдельная доработка.

### 8.5 Принудительный отзыв (УК)

- УК может отозвать доступ любого пользователя к объектам в своём ЖК
- Отзыв логируется с указанием: кто отозвал, у кого, когда, причина (текстовое поле)
- Пользователь получает push-уведомление об отзыве доступа

## 10. Отказоустойчивость

### 10.1 Поведение устройств при сбоях

| Сценарий | Поведение устройства |
|---------|---------------------|
| Устройство потеряло интернет (питание есть) | Через 90 сек прошивка активно удерживает сигнал на реле → дверь **«открыто»** (активный fail-open, см. §11.6) |
| MQTT-брокер недоступен (питание есть) | Устройство переподключается; по достижении порога 90 сек → активный fail-open |
| Сервер недоступен (питание есть) | Аналогично: активный fail-open по порогу; сервер при этом не принимает команды на открытие |
| Пропадание электропитания устройства/замка | Замок обесточен → дверь **«открыто»** (физический fail-open) |

> **Обоснование fail-open:** объект — обычный подъезд многоквартирного дома (не спецобъект). Без связи жильцы не могут открыть дверь из приложения, но должны попасть домой, поэтому при потере связи дверь открывается. Физическая безопасность/эвакуация приоритетнее защиты от несанкционированного входа.
>
> **Принятый риск (решение заказчика):** глушение или обрыв связи, равно как и обесточивание устройства, открывает дверь. Риск принят осознанно. **Митигация:** алертинг offline-устройств для УК (§19), логирование каждого события `fail_open_activated` в аудит (§16), возможность УК видеть длительно-offline точки доступа.
>
> **Восстановление:** при стабильном возврате связи (≥ 30 сек) прошивка снимает сигнал → дверь заперта, публикует статус `online`.

### 10.2 Поведение при отсутствии интернета у клиента

- Мобильное приложение: показывает кешированный список адресов, кнопки неактивны, ошибка «нет подключения»
- Веб-клиент посетителя: страница не загружается или показывает ошибку

> **Фолбэк доступа для жильца.** Если у жильца нет связи, но устройство подъезда онлайн — открыть из приложения нельзя (команда идёт через сервер). Массовый сбой (нет связи у устройства) переводит дверь в fail-open (§10.1), то есть жилец войдёт. Для случая «жилец без связи, устройство онлайн» рекомендуется предусмотреть штатный механический способ входа (ключ/домофонный ключ-таблетка) как резерв — **подтвердить с заказчиком** (вопрос эксплуатации, вне ПО).

### 10.3 Поведение сервисов при сбоях

| Сервис | При отказе |
|--------|-----------|
| Call Signaling | Звонок не устанавливается, посетителю — сообщение об ошибке |
| Snapshot Service | Фотофиксация пропускается, событие логируется без фото |
| Notification Service | Push недоставлен, звонок всё равно поступает через WebSocket |
| Audit Service | События буферизуются локально и отправляются при восстановлении |

### 10.4 Circuit Breaker

- Между сервисами — Circuit Breaker pattern (Hystrix / Resilience4j или аналог)
- При отказе Access Control Service: команды на открытие не принимаются (безопасное состояние для серверной стороны, устройство при этом в fail-open)
- Health check эндпоинт: `GET /health` на каждом сервисе

## 11. Протокол устройств (MQTT)

### 11.1 Подключение

- Протокол: **MQTT 3.1.1** или **MQTT 5.0**
- Транспорт: **MQTT over TLS (порт 8883)**
- Аутентификация устройства: **X.509 client certificate** (уникальный сертификат на устройство)
- Client ID: `device:{device.serial}`
- Keep alive: 60 секунд
- Clean session: false (persistent session)

### 11.2 Структура MQTT-топиков

```
devices/{device_id}/commands      # Сервер → Устройство (команды)
devices/{device_id}/status        # Устройство → Сервер (статус, heartbeat)
devices/{device_id}/ota           # Сервер → Устройство (обновление прошивки)
devices/{device_id}/telemetry     # Устройство → Сервер (метрики, температура)
```

### 11.3 Команды (сервер → устройство)

**Топик:** `devices/{device_id}/commands`  
**QoS:** 1 (at least once)

> **Идемпотентность на устройстве:** при QoS 1 команда может быть доставлена более одного раза. Устройство **обязано** хранить последние 20 обработанных `request_id` (TTL 60 сек) и игнорировать повторные команды с тем же `request_id`.
>
> **Свежесть команды:** устройство дополнительно проверяет `issued_at` — команда `open_relay` старше 30 секунд отбрасывается (защита от исполнения буферизованной команды «задним числом» после долгого offline, см. §11.6). Требует синхронизации времени устройства (NTP/SNTP).
>
> **Целостность:** команды защищены транспортом (MQTT over TLS + client cert + ACL, §11.1, §17.3). Для защиты от компрометации брокера рекомендуется опциональная подпись payload команды секретом устройства (проверяется прошивкой) — заложить в схему на будущее.

```json
// Открытие реле
{
  "cmd": "open_relay",
  "relay_id": 1,
  "duration_ms": 5000,
  "request_id": "uuid",
  "issued_by": "user_id",
  "issued_at": "2026-05-26T10:00:00Z"
}
```

**Доступные команды:**

| `cmd` | Описание |
|-------|---------|
| `open_relay` | Замкнуть реле на указанное время |
| `close_relay` | Принудительно разомкнуть реле |
| `reboot` | Перезагрузить устройство |
| `ping` | Проверить доступность устройства |

### 11.4 Статус (устройство → сервер)

**Топик:** `devices/{device_id}/status`  
**QoS:** 1  
**Периодичность heartbeat:** каждые 30 секунд

```json
{
  "device_id": "uuid",
  "status": "online",
  "relay_state": "closed",
  "firmware_version": "1.2.3",
  "uptime_sec": 86400,
  "rssi": -65,
  "timestamp": "2026-05-26T10:00:00Z"
}
```

**Подтверждение команды:**
```json
{
  "type": "command_ack",
  "request_id": "uuid",
  "result": "ok",
  "timestamp": "2026-05-26T10:00:01Z"
}
```

### 11.5 OTA-обновления прошивки

```
Сервер → devices/{device_id}/ota:
{
  "version": "1.3.0",
  "url": "https://ota.example.com/firmware/1.3.0.bin",
  "checksum_sha256": "abc123...",
  "mandatory": false
}

Устройство:
1. Скачивает прошивку по URL
2. Проверяет SHA-256 checksum
3. Применяет обновление (не прерывая работу реле)
4. Перезагружается
5. Публикует статус с новой версией прошивки
```

- OTA не прерывает активное реле
- При ошибке OTA — устройство остаётся на текущей версии
- Принудительное обновление (`mandatory: true`) применяется при следующем перезапуске

### 11.6 Электрическая схема и offline-поведение устройства

**Электрическая схема (замок — магнитный, fail-safe):**

- Замок питается по силовой цепи **12–15 В**, которая проходит через **нормально-замкнутые (NC) контакты реле**. В покое (катушка реле обесточена) NC-контакты **замкнуты** — замок под питанием, дверь **заперта**.
- Для открытия плата управления (ESP32) подаёт **управляющий сигнал 5 В на катушку реле** — контакты **размыкаются**, силовая цепь замка обесточивается, дверь **открыта**.
- Таким образом дверь открыта в двух случаях: (а) подан сигнал 5 В на катушку (команда открытия или активный fail-open); (б) пропало электропитание всей системы (замок обесточен физически).

**Активный fail-open при потере связи** (осознанное решение, см. §10.1):

- Критерий потери связи: нет активного соединения с MQTT-брокером **и/или** нет ответа сервера дольше порога.
- Порог: **90 секунд** без heartbeat/соединения (согласовано с периодом heartbeat 30 сек, §11.4).
- При срабатывании: прошивка **активно удерживает сигнал** на катушке реле → дверь открыта до восстановления связи.
- Защита от флаппинга: возврат в норму (снять сигнал → заперто) только после стабильного восстановления связи ≥ 30 сек.
- **Аппаратный watchdog обязателен:** зависание прошивки при живом питании не должно оставлять дверь навсегда запертой (иначе жильцы не попадут домой).
- Учесть длительное удержание катушки: нагрев/энергопотребление, выбор реле с соответствующим ресурсом.

**Свежесть буферизованных команд:**

- Буферизация команд на брокере: persistent session, TTL 5 минут.
- При переподключении устройство **проверяет свежесть** каждой команды по `issued_at`: команда `open_relay` старше **30 секунд** игнорируется (защита от открытия «задним числом» после долгого offline).
- При восстановлении связи публикует статус `online` и накопленные `command_ack`.

## 12. Аутентификация и онбординг

### 12.1 Способы аутентификации

| Пользователь | Способ |
|-------------|--------|
| Жилец / Владелец | Инвайт-ссылка (WhatsApp/Telegram) → регистрация по телефону |
| Гость / Курьер | Временный токен в ссылке (без регистрации) |
| Администратор УК | Email + пароль + **2FA (TOTP)** — обязательно (админ управляет всеми домами и жильцами тенанта) |
| Системный администратор | Email + пароль + 2FA (TOTP) |

### 12.2 Онбординг жильца

```
1.  УК/владелец создаёт запись жильца: имя, номер телефона, квартира, роль
2.  Система генерирует инвайт-ссылку: https://app.example.com/invite/{token}
    - Токен: случайный, 32 байта, однократного использования
    - TTL токена: 7 дней
3.  УК/владелец отправляет ссылку жильцу через WhatsApp или Telegram (копирует ссылку)
4.  Жилец переходит по ссылке:
    a. Если мобильное приложение установлено — deeplink открывает приложение
    b. Если нет — открывается страница с кнопкой скачивания приложения
5.  В приложении жилец вводит номер телефона (проверяется совпадение с записью)
    - Если номер не совпадает → ошибка «Номер телефона не совпадает с приглашением», предложить связаться с УК
    - Если аккаунт с таким номером уже существует → инвайт привязывается к существующему аккаунту **после** успешного OTP (см. ниже)
6.  На телефон **всегда** отправляется SMS OTP — подтверждение владения номером обязательно, даже если аккаунт уже существует (инвайт-ссылка — bearer-токен, одного её наличия недостаточно)
7.  Жилец вводит OTP — аккаунт создан/привязан, инвайт-токен аннулирован
8.  При входе в будущем: номер телефона + SMS OTP (без пароля)
```

### 12.3 Управление токенами

| Параметр | Значение |
|---------|---------|
| Access Token тип | JWT (RS256) |
| Access Token TTL | 15 минут |
| Refresh Token TTL | 30 дней |
| Refresh Token хранение | HttpOnly + Secure + SameSite=Strict cookie (мобильный — secure storage) |
| Ротация Refresh Token | При каждом обновлении (sliding window) |
| Отзыв токенов | Через refresh token blacklist (Redis) |

**JWT payload (минимальный):**
```json
{
  "sub": "user_id",
  "roles": [{"role": "resident", "apartment_id": "uuid"}],
  "iat": 1748123456,
  "exp": 1748124356
}
```

### 12.4 SMS OTP

- Провайдер: настраиваемый (Twilio, SMS.ru, локальный казахстанский провайдер)
- OTP: 6 цифр, TTL 5 минут
- Rate limiting: 3 попытки на номер за 10 минут
- Блокировка: при 5 неверных попытках — номер блокируется на 30 минут

## 13. Backend API

### 13.1 Принципы

## 16. Логи и аудит

### 16.1 Принципы

- Все события логируются в Audit Service
- Лог **append-only**: записи не редактируются и не удаляются. Хранилище — WORM/append-only (напр. S3 Object Lock в режиме Compliance или иммутабельная таблица без UPDATE/DELETE-прав у сервисов)
- **Tamper-evidence через hash-chaining:** каждая запись содержит `prev_hash` = хеш предыдущей записи в цепочке; `record_hash = SHA-256(json_canonical(record) + prev_hash)`. Разрыв цепочки (удаление/подмена записи) детектируется при верификации. Периодически публикуется «якорь» (последний `record_hash`) в отдельное защищённое хранилище
- Дополнительно запись подписывается: `HMAC-SHA256(json_canonical(record), AUDIT_SECRET_KEY)` в поле `signature` (защита от подделки при известной схеме; не заменяет hash-chain)
- Ключ `AUDIT_SECRET_KEY` хранится в KMS/Vault, доступ на подпись — только у Audit Service; ротация ключа с сохранением `kid`
- Хранение: минимум **1 год**

### 16.2 Типы событий

| Категория | Примеры событий |
|-----------|----------------|
| Доступ | door_opened, door_denied, relay_command_sent |
| Звонки | call_initiated, call_accepted, call_declined, call_timeout |

## 17. Безопасность

### 17.1 Аутентификация и авторизация

- Все API-запросы требуют валидного JWT (кроме публичных QR-страниц и гостевых ссылок)
- RBAC проверяется на сервере при каждом запросе
- Гостевые токены: одноразовые UUID, хранятся в Redis с TTL
- Refresh token rotation: при каждом обновлении старый токен инвалидируется

### 17.2 Transport Security

- HTTPS везде (TLS 1.2+, предпочтительно TLS 1.3)
- MQTT over TLS (порт 8883)
- Устройства: X.509 client certificate, уникальный на каждое устройство
- HSTS: `max-age=31536000; includeSubDomains`

### 17.3 Защита от атак

| Угроза | Мера |
|--------|------|
| Replay attack на QR | HMAC-подпись + session token одноразовый (TTL 5 мин) |
| Replay attack на команду | `request_id` уникален, проверяется на идемпотентность |
| Brute force OTP | Rate limit: 3 попытки / 10 мин, блокировка 30 мин |
| DDoS на QR-страницу | Rate limit по IP + public_id, CDN с WAF |
| CSRF | Access-JWT в заголовке (не в cookie). Refresh-token в HttpOnly-cookie защищён `SameSite=Strict` + CSRF-токен на эндпоинте `/auth/refresh` для **всех** веб-клиентов (не только admin) |
| Утечка данных | Tenant isolation, pre-signed URL для медиа |
| Перехват MQTT | TLS + client cert, минимальные MQTT ACL |
| SQL injection | Параметризованные запросы, ORM |

### 17.4 Безопасность QR-кода

- `public_id` в QR — не внутренний `id` из БД (защита от enumeration)
- HMAC-подпись с версионированием ключа через `kid`: несколько активных ключей одновременно, ротация без перепечатки статических QR; отзыв `kid` — только при компрометации (см. §5.3)
- Компрометация QR не даёт прямого доступа — нужна серверная валидация + геопроверка

### 17.5 Аудит безопасности

- Все неуспешные попытки доступа логируются
- Алерты на аномалии: множественные ошибки OTP, частые отказы геопроверки, offline устройства

## 20. Multi-tenancy

### 20.1 Модель изоляции

- Изоляция на уровне данных: каждая таблица, попадающая под tenant-изоляцию, содержит денормализованную колонку `management_company_id` (см. §2: Building, ResidentialComplex, Apartment, AccessPoint, Device, QRCode, UserApartmentRole, GuestAccess, CourierAccess, а также записи аудита §16.3). Колонка заполняется при создании и не меняется.
- PostgreSQL Row-Level Security (RLS) политики на этих таблицах фильтруют строки по `management_company_id` из контекста сессии/запроса.
- Все API-запросы автоматически фильтруются по `management_company_id` текущего пользователя
- Прямой доступ к данным другой УК невозможен даже при знании ID
- Пользователи без привязки к УК (Guest/Courier) работают только через токен-эндпоинты, которые сами резолвят `management_company_id` целевого AccessPoint и не дают перечисления данных тенанта.

### 20.2 Уровни изоляции

| Слой | Изоляция |
|------|---------|
| База данных | Общая БД, row-level security через `management_company_id` |
| API | Фильтрация на уровне сервисного слоя |
| Файловое хранилище | S3-бакеты с prefix по `management_company_id` |
| MQTT | ACL по `device_id`, устройства не могут подписаться на топики чужих устройств |
| Логи | Фильтрация в Audit Service по `management_company_id` |

### 20.3 Ограничения на уровне УК

| Лимит | Значение | Настраиваемость |
|-------|---------|----------------|
| Макс. домов | 10 000 | Да (SysAdmin) |

| Макс. пользователей | Не ограничено | — |
| Хранилище S3 | 5 ТБ | Да (SysAdmin) |

### 21.2 Согласие на фотосъёмку и обработку данных

**Где запрашивается:**
- На QR-странице посетителя: до начала звонка показывается уведомление:

> **Юридическая заметка.** ✅ **Решение (2026-07-06):** распознавание лиц (face-matching) в системе **не планируется** — обрабатывается простое фото/видео, что не является биометрией по смыслу закона. **Как сохраняется:**
- Факт согласия логируется с timestamp, IP-адресом, хешем сессии

## 23. Глоссарий

| Термин | Определение |

| **OTA** | Over-The-Air — обновление прошивки устройства по сети |

| **RBAC** | Role-Based Access Control — управление доступом на основе ролей |
| **RPO** | Recovery Point Objective — максимально допустимая потеря данных при сбое |
| **RTO** | Recovery Time Objective — максимально допустимое время восстановления |
| **SFU** | Selective Forwarding Unit — медиасервер, пересылающий медиапотоки между участниками WebRTC-сессии |

| **TLS** | Transport Layer Security — протокол шифрования транспортного уровня |
| **TOTP** | Time-based One-Time Password — одноразовый пароль на основе времени (2FA) |
