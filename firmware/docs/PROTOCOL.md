# PROTOCOL.md — MQTT-контракт устройства

**Статус:** контракт walking skeleton (по ТЗ v1.3, §11).
**Область действия:** единый контракт для Go-эмулятора (`firmware/emulator`) и будущей ESP32-прошивки (`firmware/esp32`). ESP32-прошивка обязана заменить эмулятор **без каких-либо изменений backend** — этот документ является единственным источником истины по протоколу.

> Документ написан ДО кода и описывает намерение. Расхождение реализации с этим контрактом — сигнал проблемы, а не повод молча «поправить документ».

---

## 1. Подключение

| Параметр | Значение (dev / walking skeleton) | Комментарий |
|---|---|---|
| Протокол | MQTT **3.1.1** | ТЗ §11.1 допускает 3.1.1 или 5.0; skeleton фиксирует 3.1.1 (совместимо с paho и ESP32-клиентами) |
| Брокер | `tcp://localhost:1883` (EMQX 5) | **dev: БЕЗ TLS, anonymous auth.** Прод: MQTT over TLS :8883 + X.509 client certificate на устройство — ТЗ §11.1, §17.2. В скоуп skeleton не входит |
| Client ID | `device:{serial}` → dev: `device:EMU-001` | ТЗ §2.2.6, §11.1 |
| Clean Session | `false` (persistent session) | Брокер буферизует QoS1-команды, выданные пока устройство offline (ТЗ §11.6) |
| QoS | 1 (at least once) — все топики | Следствие: возможна повторная доставка → идемпотентность обязательна (§5.1) |
| Keep alive | **30 секунд** | ⚠ ОТКЛОНЕНИЕ от ТЗ §11.1 (60 сек) — обоснование ниже |

### Обоснование keepalive 30с (отклонение от ТЗ §11.1)

Брокер детектирует обрыв соединения через ~1.5×keepalive. При keepalive 60с детекция занимает до 90с — ровно на границе порога fail-open (90с, ТЗ §10.1/§11.6) и порога offline (90с, ТЗ §2.2.6), что делает тайминги edge-сценариев недетерминированными. При keepalive 30с обрыв детектируется за ≤45с — заведомо внутри порога 90с и согласовано с периодом heartbeat 30с.

**Для ESP32 значение пересмотреть** (энергопотребление/трафик GSM), но менять его можно только синхронно с порогами fail-open/offline: инвариант `1.5 × keepalive < 90с` должен сохраняться.

## 2. Топики

`{device_id}` — **внутренний UUID устройства** (не serial): в skeleton это фикстура `66666666-6666-6666-6666-666666666666` (см. `docs/skeleton/architecture.md`, раздел «Фиксированные UUID фикстуры»).

| Топик | Направление | QoS | Payload |
|---|---|---|---|
| `devices/{device_id}/commands` | сервер → устройство | 1 | Команда `open_relay` (§3) |
| `devices/{device_id}/status` | устройство → сервер | 1 | 3 типа сообщений: `heartbeat`, `command_ack`, `event` (§4) |

Топики `.../ota` и `.../telemetry` (ТЗ §11.2), команды `close_relay` / `reboot` / `ping` (ТЗ §11.3) — **вне скоупа walking skeleton** и в контракт не входят.

---

## 3. Команды: `devices/{device_id}/commands`

### 3.1 `open_relay` — единственная команда skeleton

```json
{
  "cmd": "open_relay",
  "relay_id": 1,
  "duration_ms": 5000,
  "request_id": "7f9c24e5-0d2b-4a1a-9b6e-3f8a2c5d1e07",
  "issued_by": "resident:33333333-3333-3333-3333-333333333333",
  "issued_at": "2026-07-07T10:00:00Z"
}
```

| Поле | Тип | Обязательное | Описание |
|---|---|---|---|
| `cmd` | string | да | Всегда `open_relay`. Неизвестный `cmd` → устройство игнорирует, пишет warning в локальный лог, ack не публикует |
| `relay_id` | int | да | В skeleton всегда `1` |
| `duration_ms` | int | да | Время удержания реле в состоянии `open`. Дефолт backend: `5000` |
| `request_id` | string (UUID v4) | да | Уникален на каждую команду. Ключ идемпотентности (§5.1) и корреляции с `command_ack` и аудитом |
| `issued_by` | string | да | Кто выдал команду. Свободный формат, устройством не интерпретируется, транслируется в аудит backend'ом |
| `issued_at` | string (ISO 8601, UTC, суффикс `Z`) | да | Момент выдачи команды сервером. Основа проверки свежести (§5.2) |

## 4. Статус: `devices/{device_id}/status`

В один топик устройство публикует **три типа** сообщений. Backend различает их по полю `type` (отсутствие `type` = heartbeat).

### 4.1 `heartbeat`

Публикуется: **каждые 30 секунд** + **немедленно при любой смене `relay_state`** (closed→open, open→closed) — именно по внеочередным heartbeat наблюдается цикл реле (AC6).

```json
{
  "device_id": "66666666-6666-6666-6666-666666666666",
  "status": "online",
  "relay_state": "closed",
  "firmware_version": "emu-0.1.0",
  "uptime_sec": 86400,
  "rssi": -50,
  "timestamp": "2026-07-07T10:00:00Z"
}
```

| Поле | Тип | Описание |
|---|---|---|
| `device_id` | string (UUID) | Внутренний UUID устройства |
| `status` | string | Всегда `online` (устройство публикует, только пока живо; `offline` выводится backend'ом по отсутствию heartbeat) |
| `relay_state` | string | `closed` — реле в покое, NC-контакты замкнуты, замок под питанием, дверь **заперта**; `open` — катушка под сигналом, замок обесточен, дверь **открыта** (ТЗ §11.6) |
| `firmware_version` | string | Эмулятор: `emu-x.y.z`; ESP32: semver прошивки |
| `uptime_sec` | int | Секунд с запуска устройства |
| `rssi` | int | Уровень сигнала; эмулятор публикует фиктивное значение (`-50`) |
| `timestamp` | string (ISO 8601 UTC) | Момент формирования |

Backend на каждый heartbeat обновляет Redis-ключ presence `device:online:{device_id}` (TTL 90с) и `devices.last_seen_at` в Postgres. Статус `online`/`offline` — **производное** от наличия ключа, не колонка БД.

### 4.2 `command_ack`

Публикуется в ответ на **каждую** полученную команду `open_relay` — включая дубликаты и отвергнутые.

**Успех** — публикуется немедленно после актуации (перевода реле в `open`); завершение цикла наблюдается по внеочередному heartbeat с `relay_state: "closed"`:

```json
{
  "type": "command_ack",
  "request_id": "7f9c24e5-0d2b-4a1a-9b6e-3f8a2c5d1e07",
  "result": "ok",
  "timestamp": "2026-07-07T10:00:01Z"
}
```

**Дубликат** (повторная доставка QoS1 или ручной replay) — актуации НЕТ, ack повторяется с флагом:

```json
{
  "type": "command_ack",
  "request_id": "7f9c24e5-0d2b-4a1a-9b6e-3f8a2c5d1e07",
  "result": "ok",
  "duplicate": true,
  "timestamp": "2026-07-07T10:00:05Z"
}
```

**Отказ по свежести** (stale) — актуации НЕТ:

```json
{
  "type": "command_ack",
  "request_id": "7f9c24e5-0d2b-4a1a-9b6e-3f8a2c5d1e07",
  "result": "rejected",
  "reason": "stale",
  "timestamp": "2026-07-07T10:01:40Z"
}
```

| Поле | Тип | Описание |
|---|---|---|
| `type` | string | Всегда `command_ack` |
| `request_id` | string (UUID) | Из исходной команды |
| `result` | string | `ok` \| `rejected` |
| `reason` | string, опц. | Только при `result: "rejected"`. В skeleton единственное значение: `stale` |
| `duplicate` | bool, опц. | `true` — команда с этим `request_id` уже была обработана ранее; повторной актуации не было, `result` повторяет исходный. При первом исполнении поле отсутствует |
| `timestamp` | string (ISO 8601 UTC) | — |

### 4.3 `event`

Асинхронные события устройства (не привязаны к `request_id`).

```json
{
  "type": "event",
  "event": "fail_open_activated",
  "timestamp": "2026-07-07T10:05:00Z"
}
```

| `event` | Когда |
|---|---|
| `fail_open_activated` | Порог потери связи достигнут, реле переведено в `open` (§5.3) |
| `fail_open_deactivated` | Связь стабильна ≥30с, реле возвращено в `closed` (§5.4) |

**Буферизация offline (обязательно):** `fail_open_activated` по определению возникает БЕЗ связи с брокером. Устройство обязано сохранить событие в локальный буфер (эмулятор — в памяти; ESP32 — допустимо RTC/NVS) и опубликовать его **сразу после реконнекта**. Иначе fail-open никогда не попадёт в аудит (AC10/AC14). Порядок публикации после реконнекта: буферизованные `event` (в порядке возникновения, с исходными `timestamp`) → текущий heartbeat → далее обычный режим.

## 5. Правила устройства (обязательные инварианты)

### 5.1 Идемпотентность

- Устройство хранит буфер последних **20** обработанных `request_id`, каждая запись с TTL **60 секунд** (ТЗ §11.3).
- Команда с `request_id`, уже присутствующим в буфере → **актуации нет**, публикуется повторный `command_ack` с `duplicate: true` и исходным `result`.
- Вытеснение: при переполнении (>20) удаляется самая старая запись; по TTL — запись исчезает через 60с.
- Повтор с тем же `request_id` **после** вытеснения из буфера будет обработан как новая команда — к этому моменту его отсечёт проверка свежести (§5.2), т.к. TTL буфера (60с) > порога свежести (30с).

### 5.2 Свежесть (anti-stale)

- Для каждой команды: `now − issued_at > 30 секунд` → команда **отвергается**: `command_ack {result: "rejected", reason: "stale"}`, актуации нет, в локальный лог пишется запись `stale_command_rejected` (AC8).
- **Допуск на рассинхрон часов: ±5 секунд** — фактический порог отбрасывания лежит в интервале 25–35с; тесты не должны проверять границу точнее ±5с.
- Назначение: буферизованная брокером команда (persistent session) не должна открыть дверь «задним числом» после восстановления связи (ТЗ §11.6).
- Команда с `issued_at` в **будущем** более чем на 30с также отвергается как `stale` (симметричная защита от кривых часов).
- Требование к часам: ESP32 — синхронизация NTP/SNTP (ТЗ §11.3); эмулятор использует системные часы хоста (совпадают с часами backend — там же).

### 5.3 Fail-open (активный, по потере связи)

- Критерий потери связи: нет живого соединения с MQTT-брокером (ТЗ §10.1/§11.6).
- Порог: **≥90 секунд** непрерывного отсутствия связи → реле переводится в `open` и **активно удерживается** до восстановления.
- В момент срабатывания событие `fail_open_activated` кладётся в offline-буфер (§4.3).
- Отсчёт 90с начинается с момента обрыва соединения (для эмулятора — callback потери соединения paho + фоновый таймер; успешный реконнект до порога сбрасывает отсчёт).

### 5.4 Восстановление после fail-open (гистерезис)

- Реле возвращается в `closed` только после **≥30 секунд непрерывно стабильной** связи с брокером после реконнекта.
- Реконнект, оборвавшийся раньше 30с, **не закрывает** реле; таймер стабильности сбрасывается и стартует заново при следующем реконнекте (защита от флаппинга, ТЗ §11.6).
- При возврате в `closed` публикуются: `fail_open_deactivated` + внеочередной heartbeat `relay_state: "closed"`.

### 5.5 Цикл реле по команде

Валидная (не дубликат, свежая) команда `open_relay`:

1. Реле → `open`; немедленный heartbeat `relay_state: "open"`.
2. Публикуется `command_ack {result: "ok"}`.
3. Через **ровно `duration_ms`** (допуск ±100 мс для эмулятора) реле → `closed`; немедленный heartbeat `relay_state: "closed"` (AC6).
4. Новая команда (другой `request_id`), пришедшая пока реле `open`, перезапускает таймер удержания от нового момента («последняя команда выигрывает»); ack публикуется штатно.
5. Если активен fail-open (§5.3), команда подтверждается (`result: "ok"`), но реле уже `open` и остаётся `open` — fail-open имеет приоритет над таймером `duration_ms`.

## 7. Безопасность: dev vs прод

| Аспект | dev (walking skeleton) | прод (ТЗ §11.1, §17.2, §17.3) |
|---|---|---|
| Транспорт | `tcp://localhost:1883`, **без TLS** | MQTT over TLS :8883 |
| Аутентификация | anonymous (EMQX dev-режим) | X.509 client certificate, уникальный на устройство |
| ACL | нет | минимальные ACL: устройство видит только свои топики `devices/{id}/...` |
| Подпись payload команды | нет | опциональная HMAC-подпись секретом устройства (закладка ТЗ §11.3) |

Переход dev → прод не меняет форматы payload и правила §5 — меняется только транспортный слой подключения.
