# Итоговый отчёт — Walking skeleton QR-домофона

**Дата:** 2026-07-11 · **Репозиторий:** github.com/VVWIZ/QR_DOMofon · **ТЗ:** v1.3

Тонкий вертикальный срез системы по ТЗ v1.3, проверяющий рискованные связки:
QR → WebRTC-звонок (LiveKit) → открытие двери → MQTT → эмулятор реле, + аудит,
presence, fail-open. Реализован через оркестратор `/build` (этапы 1–12).

## 3. Ключевые находки и фиксы

**Безопасность (этап 8):** M1 — гейт accept→open (дверь открывает жилец, не владелец QR);
L1 — HTTP-таймауты + лимиты тел; L2 — UUID-валидация `aid`. M2 (SSE без auth) оставлен в
прод-гэпах осознанно.

**Code review (этап 9):** блокеров нет; мелочи (тип `last_seen_at`, код `CALL_NOT_ACCEPTED`
на фронте) поправлены.

**Живая интеграция — окружение (этап 7):**
- `emqx/emqx:5` → `5.8.6` (тег `:5` не существует).
- LiveKit `1.9.0` → `1.10.0` (клиент 2.20.1 требует protocol 17 / `/rtc/v1`).
- **LiveKit `node_ip` = LAN-IP** (не 127.0.0.1): виртуальные адаптеры WSL/Docker отравляли
  ICE-кандидаты, publisher-медиа зависало. + нативный запуск LiveKit на Windows (План Б).
- Окружение: требовалась VT-x в BIOS + WSL2-ядро; Kaspersky обходится `-ldflags="-s -w"`
  для эмулятора. Всё задокументировано в `RUNBOOK.md`.

## 5. Анти-скоуп (сознательно вне скелета)

RBAC/роли, гости/курьеры, гео-проверка, фотофиксация, push, SMS OTP, мобильное приложение,
реальное железо (ESP32 — шаг 2), hash-chain аудита, OTA, TLS/X.509 для MQTT, rate-limit,
RLS/multi-tenancy enforcement, админки, 2FA. Заготовки под продукт есть (напр.
`management_company_id` во всех tenant-таблицах).

**Прод-гэпы к закрытию до продакшена:** аутентификация SSE + скоуп по квартире; auth/RBAC
на все эндпоинты; MQTT TLS+X.509+ACL (иначе прямое открытие через брокер); QR-секрет в KMS;
короткий TTL LiveKit-токена; rate-limit; RLS; `sslmode=require`.
