# Итоговый отчёт — Walking skeleton QR-домофона

**Дата:** 2026-07-11 · **Репозиторий:** github.com/VVWIZ/QR_DOMofon · **ТЗ:** v1.3

Тонкий вертикальный срез системы по ТЗ v1.3, проверяющий рискованные связки:
QR → WebRTC-звонок (LiveKit) → открытие двери → MQTT → эмулятор реле, + аудит,
presence, fail-open. Реализован через оркестратор `/build` (этапы 1–12).

## Инкремент auth/RBAC (2026-07-12)

Аутентификация и RBAC поверх skeleton. Дизайн — [auth.md](auth.md), контракт — [api.md](api.md).

**Скоуп:** жилец/владелец (телефон → SMS OTP мок → JWT RS256) + УК-админ (email+пароль
bcrypt + TOTP 2FA); защита эндпоинтов + RBAC (apartment-membership на accept/open,
роль-чек, admin-скоуп по mc). JWT access 15м + refresh 30д (Redis whitelist, ротация).

**Проверено на живой системе (этап 7):**

| Проверка | Результат |
|---|---|
| Аноним на защищённые (`/devices`, `/access/open`) | 401 ✓ |
| OTP-логин жильца → JWT + refresh-cookie, `/me` | ✓ |
| RBAC: жилец → admin-only, admin → resident-only | 403 (обе стороны) ✓ |
| Admin-логин email+пароль+**TOTP** → admin-скоуп `/devices`,`/audit` по mc | ✓ |
| Полный флоу с токеном: initiate→accept→open→**реле** | ✓ |
| Refresh-ротация по cookie; неверный OTP → 401 | ✓ |
| Регрессия: визитёрский `qr/validate` public | ✓ |

**Тесты:** 30 юнит-тестов auth зелёные (JWT/OTP/TOTP/bcrypt/refresh/RBAC/middleware);
regression skeleton не затронут.

**Security-ревью:** блокеров нет; обход RBAC, alg-confusion, утечка токена/чужой УК —
не найдены. Применены: `AUTH_DEV_MODE=false` (secure-by-default), constant-time OTP.

**Известные minor (митигированы, к проду):** неатомарный refresh-Rotate (митигация —
single-flight на фронте); refresh без re-check ролей в БД (митигация — access 15м);
admin timing-enumeration по email; PII (phone) в аудите `otp_failed`; modulo-bias
генерации OTP (ничтожно).

**Прод-гэпы инкремента (вне скоупа):** реальный SMS-провайдер, CSRF-токен на /auth/refresh
(пока SameSite=Strict), одноразовый SSE-тикет вместо `?token=`, онбординг жильцов,
SystemAdmin, гости/курьеры, полная админка УК (только вход). Dev-keypair RS256 закоммичен
(dev-only, в проде — из KMS).

## Инкремент «онбординг + гранты доступа» (2026-07-14)

Кто кого заводит и как жилец получает доступ. Контракт — [api.md](api.md) §Онбординг,
дизайн входа — [auth.md](auth.md) §5.

**Скоуп (утверждён с заказчиком):** УК создаёт **владельцев** и раздаёт доступ на
**калитки/шлагбаумы**, видит всех жильцов своей УК (но жильцов НЕ создаёт); **владелец**
приглашает **жильцов** в свою квартиру; приём инвайта **без СМС** (bearer-ссылка);
**прямое открытие** калитки/шлагбаума по гранту, без звонка (ТЗ §4.7). Доставка инвайта —
мок (ссылка в ответе API).

**Модель данных:** миграция `0005_onboarding` — `invites` (одноразовые, хранится только
`sha256(token)`, TTL 7 дней) + `user_access_grants` (UNIQUE user×точка); сид калитки
`Калитка двора` (`eeeeeeee-…`, gate) + устройство EMU-002.

**Что на вебе, что в мобильном (ТЗ §14).** Жилец/владелец в продукте работают через
**мобильное приложение**; браузер — у гостя. Поэтому на вебе сделана только **УК-консоль**
(`/admin`), а резидентские флоу онбординга — **API-first** (потребитель — мобильный клиент).
Одноразовый резидентский веб-UI сознательно не писался.

### Живой прогон (этап 5)

| Проверка | Результат |
|---|---|
| УК создаёт владельца → инвайт → приём → **вход без OTP** | ✓ kind=owner, роль на кв. |
| Владелец приглашает жильца в свою кв. → приём | ✓ kind=resident |
| Владелец приглашает в **чужую** квартиру | ✓ 403 `FORBIDDEN` |
| Повторный приём использованного инвайта | ✓ 404 `INVITE_INVALID` (атомарный `used_at`) |
| УК выдаёт грант на калитку (новому / уже в УК) | ✓ инвайт (201) / `granted:true` (200) |
| `GET /access/points` у грантованного | ✓ `Калитка двора`, `online:true` |
| **Прямое открытие калитки** `open-point` | ✓ 200 → MQTT → **реле EMU-002 сработало** (request_id сквозной: API → эмулятор → аудит) |
| Жилец без гранта / грант на другую точку | ✓ 403 обе стороны, `/points` пуст |
| Список жильцов УК | ✓ агрегация квартир+грантов, скоуп по `mc_id` |

### Security-ревью (этап 6) — **два блокера найдены и закрыты**

1. **Эскалация привилегий (HIGH).** Приём инвайта искал пользователя по телефону без учёта
   `kind`, а токен выпускался с `kind` найденного. Инвайт на телефон `mc_admin` выдавал
   **админский токен** — а через админа другой УК означал **захват чужого тенанта**.
   *Фикс:* поиск/создание только `kind ∈ {resident, owner}`; телефон админа → `INVITE_INVALID`.
2. **Угон аккаунта (MEDIUM-HIGH).** Приём инвайта на телефон существующего пользователя выдавал
   сессию за него **со всеми его ролями** (включая другие УК) — т.е. любой, зная телефон, мог
   выпустить себе чужую сессию. *Фикс:* авто-вход только **новому** пользователю; существующему —
   привязка без сессии (`{linked, login_required}`), вход обычным OTP.

Также по ревью: аудит онбординга (`invite_created` / `invite_accepted` / `access_grant_created`,
без утечки токенов); owner-инвайт повышает `kind` **и роль** (resident→owner), resident-инвайт
владельца не понижает; нормализация телефона к `+E.164` (`8XXX` == `+7XXX` — иначе дубли
аккаунтов); «чужая УК» и «не найдено» отвечают одинаково (закрыт enumeration чужих объектов).

Все шесть фиксов перепроверены на живой системе (эскалация → 404; угон → `login_required`;
новый юзер входит; номер схлопывается; аудит пишется, утечек токенов 0; enumeration неразличим;
open-point → реле).

**Держит по итогам ревью:** весь SQL параметризован · секрет инвайта только как SHA-256, в логи
не попадает · токен 256 бит + rate-limit на accept · атомарный claim инвайта (гонка проверена) ·
RBAC (ownership / mc-скоуп / грант) — три живых 403.

**Тесты:** onboarding 14 (токен, валидность инвайта, ownership, нормализация), access 3
(open-point: грант / нет гранта / offline), auth 30 — зелёные; регрессий скелета нет.

**Прод-гэпы инкремента:** OTP-подтверждение телефона на `invite/accept` (снимет оба ограничения
приёма); мобильное приложение (резидентские экраны); отзыв/переотправка инвайта; отзыв гранта;
гости/курьеры; SystemAdmin.

> **Уточнение по доставке инвайта (2026-07-16):** «мок доставки» переоценён как **целевой дизайн**.
> По решению заказчика приглашающий сам пересылает секрет-ссылку адресату (SMS/мессенджер), поэтому
> серверная интеграция с провайдером рассылки не требуется — это не гэп.

## Инкремент A — подъезды, ФИО, композитный инвайт (2026-07-16)

Первый из двух инкрементов «форма создания инвайта» (A = данные + УК-консоль; B = гости — отдельно).
Дизайн-стратегия верифицирована `@architect` (Fable, expand-contract). Контракт — [api.md](api.md)
§Онбординг, схема — [architecture.md](architecture.md) §3.

**Скоуп:** (1) **подъезды** (сущность `entrances`: дом→подъезд→квартира); (2) **ФИО** (`full_name`,
обязательна на вводе); (3) **композитный инвайт УК** — одна ссылка заводит владельца квартиры + сразу
N грантов на калитки/шлагбаумы; (4) **каталог** `GET /admin/catalog` для выпадашек формы;
(5) форма УК с выбором дом/подъезд/квартира и чекбоксами доступов.

**Стратегия миграции (0006):** expand-фаза expand-contract — всё аддитивно, работающий скелет не
затронут. `entrances` + nullable `entrance_id` с композитным FK `(entrance_id, building_id)` (гард
чужого дома); `building_id` сохранён → `property.ResolveByPublicID` не изменён. Composite-инвайт —
дочерняя `invite_access_points`, атомарный `used_at`-claim и `invites_shape` не тронуты. Приём доп.
грантов — одной set-based `INSERT..SELECT ORDER BY` (без Go-цикла → без дедлока).

**Живой прогон (этап A4):**

| Проверка | Результат |
|---|---|
| Каталог: дерево дом→подъезд→квартира + точки | ✓ |
| Композитный инвайт (квартира + грант на калитку) → приём | ✓ kind=owner, роль owner, грант выдан |
| Кэп ≤ 20 точек | ✓ 400 `VALIDATION_ERROR` |
| Композит с чужой/несуществующей точкой | ✓ 400, **инвайт не создан** (атомарно) |
| Композит с точкой-подъездом (не gate/barrier) | ✓ 400 |
| Дедуп (одна точка ×3) | ✓ 1 грант |
| Композит существующему пользователю | ✓ привязка есть, **сессии нет** (`login_required`) |
| ФИО дозаполняется только если пусто; непустое не перезатирается | ✓ (UTF-8 roundtrip идентичен) |
| Аудит `invite_accepted` с `granted_points` | ✓ |
| **Регрессия скелета** после backfill entrance (QR-валидация + `calls/initiate`) | ✓ резолвит квартиру |

**Security-ревью A:** новых дыр нет. Главный риск композита — опечатка в телефоне (существующий
пользователь молча получает owner + N грантов) — митигирован кэпом ≤20 + дедупом + полным списком
точек в аудите, и наследует правило «сессия только новому». Set-based вставка грантов детерминирована
(`ORDER BY`) — без дедлока с конкурентным `AttachAccessGrant`.

**Тесты:** onboarding 22 (+ нормализация телефона, кэп/дедуп композита), access 3, auth 30 — зелёные.
`go build/vet/gofmt`, `tsc`, `vite build` — чисто.

**Прод-гэпы A / вынесено в B:** гости/курьеры (ТЗ §2.2.8–2.2.9, `/g/{token}`, ≤2 дня); перевод
`ResolveByPublicID` на подъезды (инкремент звонков); отзыв инвайта; contract-фаза `entrance_id NOT NULL`.

## Инкремент B — гостевой доступ (2026-07-16)

Второй инкремент «формы создания инвайта»: временный доступ для гостя из ТЗ §2.2.8/§3.1.5/§4.6.
Дизайн верифицирован `@architect` (Fable). Контракт — [api.md](api.md) §Гостевой доступ, схема —
[architecture.md](architecture.md) §3 (0007).

**Скоуп:** гость **без аккаунта** по ссылке `/g/{token}`, окно ≤ 2 дней; создаёт владелец или жилец с
делегированным `can_create_guests`; точки гостя ⊆ доступных создателю (подъезд квартиры + его гранты
на калитки/шлагбаумы); гостевая **веб-страница** с кнопками открытия; делегирование, отзыв. Курьеры
(§2.2.9) — отложены.

**Ключевые решения:**
- **Производный доступ, динамический.** Право переспрашивается на КАЖДОМ открытии: подъезд — по
  членству создателя в квартире, gate/barrier — по действующему `user_access_grants` создателя. УК
  изъяла грант → гость мгновенно теряет точку (проверено вживую). Отзыв `can_create_guests` уже
  созданных гостей не убивает (право управляет созданием, не жизнью).
- **Переиспользование машинерии открытия.** Из `access.OpenPoint` извлечён `OpenResolved(gp, actor,
  meta)` (presence→MQTT→аудит) — гость дёргает тот же путь через `guests.DoorOpener` + адаптер, ноль
  дублирования и ноль знания access о гостях (3 теста access не тронуты).
- `can_create_guests` проверяется по БД (не по claims): свежее делегирование действует сразу,
  отозванное — сразу перестаёт.
- Токен — как инвайт (hash-only), окно в БД (CHECK) + сервисе; `no-store`/`no-referrer` на `/g/*`;
  открытие под 10/мин.

**Живой прогон (этап B4):**

| Проверка | Результат |
|---|---|
| Happy-path: создать (24ч) → `/g/{token}` → открыть калитку + подъезд | ✓ оба реле |
| **Производность**: УК изъяла грант владельца → гость теряет калитку / хранит подъезд | ✓ **403 / 200** |
| Жилец без права создаёт гостя | ✓ 403 |
| Окно > 48ч | ✓ VALIDATION_ERROR |
| Точка вне доступа создателя | ✓ 403 |
| Отзыв гостя → страница/открытие | ✓ 410 GUEST_EXPIRED |
| Невалидный/чужой токен | ✓ 404 GUEST_INVALID |
| Делегирование → жилец с правом создаёт | ✓ 200 → 201 |
| Токен в аудите/логах | ✓ 0 утечек |

**Security-ревью B:** новых дыр нет. Риск bearer-ссылки принят заказчиком; митигирован TTL≤2д +
revoke + hash-only + заголовки + rate-limit. Производный переспрос закрывает «гость переживает права
создателя». Enumeration невозможен (256-бит токен + единый 404 + лимит).

**Тесты:** guests 10 (окно/отзыв/границы), access 3 (не тронуты), onboarding 22, auth 30 — зелёные.
`go build/vet/gofmt`, `tsc`, `vite build` — чисто.

## Инкремент C — Объект (ЖК) + платформенная админка SystemAdmin (2026-07-16)

Первый из трёх инкрементов надстройки (C = данные+SystemAdmin; D = редизайн УК-консоли в матрицу
доступа; E = авто-открытие точек по расписанию). Дизайн верифицирован `@architect` (Fable).

**Скоуп:** (1) сущность **Объект** (`sites`, ЖК/отдельный объект) между УК и домом — иерархия
Платформа→УК→Объект→дом→подъезд→квартира; калитки/шлагбаумы авторитетны по объекту; (2) роль
**`system_admin`** — отдельная админка `/system`, не ограничена УК; (3) CRUD платформы: создание УК,
УК-админов, объектов, домов, подъездов + перевешивание дома.

**Ключевое решение (архитектор):** `site_id` сразу `NOT NULL` — в системе НЕТ код-путей, создающих
mc/дома/точки (всё seed-only), поэтому expand-contract без nullable-долга. Работающие резолверы
(`ResolveByPublicID`, гранты, гости) не тронуты: `building_id` у существующих точек сохранён.

**Разграничение уровней (критично):** `system_admin` НЕ проходит `RequireAdmin` — иначе admin-выборки
со скоупом по пустому `mc_id` дали бы утечку/пустоту. `IsAdmin()` строго `mc_admin`; отдельный
`RequireSystemAdmin`. Вход — общий `/auth/admin/login`, различение по kind.

**Живой прогон (этап C4):**

| Проверка | Результат |
|---|---|
| sysadmin: создать УК → объект → дом → подъезд + УК-админ | ✓ |
| Созданный УК-админ входит по серверному TOTP (kind=mc_admin, mc scoped) | ✓ |
| **Разграничение**: `system_admin` → `/admin/*` | ✓ 403 |
| **Разграничение**: `mc_admin` → `/system/*` | ✓ 403 |
| Анти-спуфинг: перевесить дом на объект чужой УК | ✓ 400 (композитный FK) |
| Анти-эскалация: инвайт на телефон `system_admin` | ✓ 404 INVITE_INVALID |
| Дубль имени УК/объекта | ✓ VALIDATION_ERROR |
| Регрессия скелета: QR-валидация после 0008 | ✓ 200 |

**Security-ревью C:** новых дыр нет. Главный риск (смешение уровней) закрыт раздельными middleware
+ 4 юнит-теста. TOTP-секрет нового админа не логируется (otpauth один раз). mc из родителя
(анти-спуфинг). Прод-риск generic-backfill «Объект по умолчанию» митигирован `PATCH building.site_id`.

**Тесты:** auth 34 (+4 разграничения уровней), guests 10, onboarding 22, access 3 — зелёные.
`go build/vet/gofmt`, `tsc`, `vite build` — чисто.

**Прод-гэпы C / вынесено:** редизайн УК-консоли в матрицу доступа + отзыв гранта → **D**;
авто-открытие по расписанию → **E**; impersonation (system_admin → консоль УК); глобальный аудит
sysadmin (ТЗ §16.3); contract-фаза `building_id=NULL` у site-level точек; RLS.
